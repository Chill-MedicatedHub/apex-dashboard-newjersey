# Chill Medicated — New Jersey Sales Dashboard

A self-contained sales dashboard for Chill Medicated's New Jersey operations.
Data is pulled from the **LeafLink V2 API** (using a Developer Options token)
and rendered as KPIs, charts, and tables.

```
apex-dashboard-newjersey/
├── .github/workflows/
│   └── refresh-data.yml   ← runs scraper every 15 min on GitHub's servers
├── scraper.py             ← LeafLink → sales_data.json
├── refresh_loop.py        ← (optional) local refresh loop, for dev
├── dashboard.html         ← the dashboard UI
├── sales_data.json        ← generated by the workflow (don't edit by hand)
├── .env.example           ← template for local development only
└── requirements.txt
```

The dashboard is served by **GitHub Pages**.
The data is refreshed by **GitHub Actions** every 15 minutes.
You don't need to install anything on your own computer.

---

## One-time setup

### 1. Add your LeafLink token as a GitHub Secret

This is the only secret the scraper needs, and GitHub will store it encrypted.

1. Go to your repo on GitHub
2. Click **Settings** (tab near top of repo)
3. In the left sidebar: **Secrets and variables** → **Actions**
4. Click the green **"New repository secret"** button
5. **Name:** `LEAFLINK_TOKEN`
6. **Value:** paste your LeafLink API token
7. Click **"Add secret"**

The token is now encrypted at rest and only injected into workflow runs as an
environment variable. It's never visible in logs, never committed to the repo,
and never visible to anyone — including you — once saved.

### 2. Enable GitHub Pages (if not already)

1. Repo → **Settings** → **Pages**
2. Under "Build and deployment" → Source: **Deploy from a branch**
3. Branch: **main**, folder: **/ (root)**
4. Click **Save**

Your dashboard will be live at:
`https://<your-username>.github.io/<repo-name>/dashboard.html`

### 3. Trigger the first data refresh

The workflow runs every 15 minutes automatically, but you can kick off the
first one immediately:

1. Repo → **Actions** tab (top of the page)
2. Click **"Refresh sales data"** in the left sidebar
3. Click the **"Run workflow"** dropdown on the right
4. Click the green **"Run workflow"** button

Wait ~1 minute. The workflow run will appear in the list. Click it to watch
the live log. When it finishes (green checkmark), `sales_data.json` is in
the repo and the dashboard will render data on next page load.

## How it works

```
┌──────────────────────────────────────────────────────────────────┐
│  Every 15 minutes:                                               │
│                                                                  │
│  GitHub Actions runner                                           │
│    │                                                             │
│    ├─→ checks out the repo                                       │
│    ├─→ installs Python + requirements.txt                        │
│    ├─→ reads LEAFLINK_TOKEN from Secrets                         │
│    ├─→ runs scraper.py                                           │
│    │     │                                                       │
│    │     └─→ calls LeafLink V2 API                               │
│    │           └─→ normalizes to sales_data.json                 │
│    │                                                             │
│    └─→ git commits sales_data.json (if changed) and pushes       │
│                                                                  │
│  GitHub Pages auto-rebuilds the site on each push.               │
│  Dashboard fetches the new sales_data.json on next page load.    │
└──────────────────────────────────────────────────────────────────┘
```

## Configuration

All runtime knobs live in the workflow file (`.github/workflows/refresh-data.yml`):

| Variable              | Default              | Purpose                                            |
| --------------------- | -------------------- | -------------------------------------------------- |
| `LEAFLINK_TOKEN`      | *(from Secrets)*     | API token                                          |
| `LEAFLINK_DAYS_BACK`  | `365`                | How much order history to pull each run            |
| `LEAFLINK_STATE`      | `NJ`                 | State to filter to                                 |
| `LEAFLINK_DOMAIN`     | `app.leaflink.com`   | Use `www.sandbox.leaflink.com` for the test env    |

To change them, edit the `env:` block in the workflow YAML and commit.

To change refresh frequency, edit the cron line:
- `*/15 * * * *` — every 15 minutes (default)
- `*/30 * * * *` — every 30 minutes
- `0 * * * *` — every hour, on the hour
- `0 */4 * * *` — every 4 hours

## Troubleshooting

### Workflow run shows a red ❌

Click the failed run → expand the failing step → read the log.

Most common causes:

- **`ERROR: LeafLink rejected the request (status 401)`** → token is wrong or
  expired. Generate a new one in LeafLink → Developer Options, update the
  `LEAFLINK_TOKEN` secret in repo Settings.
- **`ERROR: LeafLink returned 403 (forbidden)`** → token exists but lacks
  permission. The user who generated it needs "Manage Orders Received"
  and "Can Access Developer Options" permissions in LeafLink.
- **`Could not write sales_data.json`** → very rare; usually means the
  Actions runner ran out of disk. Re-run the workflow.

### Dashboard shows "No data yet"

The workflow hasn't run successfully yet. Go to the Actions tab and check
the most recent run. If it succeeded but the dashboard still shows no data,
do a hard refresh (Ctrl+Shift+R) — your browser might be caching the old
empty state.

### Workflow runs successfully but dashboard data is stale

GitHub Pages caches aggressively. Wait 2-3 minutes after a successful
workflow run, then hard-refresh.

## Local development (optional)

If you want to run the scraper on your own computer (e.g. to debug, or to
pull data manually outside the schedule), see `.env.example` and:

```
pip install -r requirements.txt
cp .env.example .env       # then edit .env and paste your token
python scraper.py
python -m http.server 8000
# Open http://localhost:8000/dashboard.html
```

This is **not required** for normal operation — the Action handles everything.

## Security notes

- Your LeafLink token is stored as an encrypted GitHub Secret. It's never
  visible in logs, commits, or to anyone with read access to the repo.
- `.gitignore` excludes `.env` so you can't accidentally commit a token
  from local development.
- The repo is public, which means `sales_data.json` is public too. If your
  sales data is sensitive, make the repo private (GitHub Pages on private
  repos requires GitHub Pro or higher).
- If the token is ever exposed, revoke it immediately in LeafLink and
  generate a new one.
